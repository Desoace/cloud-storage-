ABOUT The Builder

Built by Dhruv.
For people who care
about privacy.

This vault started as a personal project — a frustration with file-sharing tools that log everything, keep your data forever, and hide their encryption behind vague marketing.

So I built my own. Every file that passes through dhruv.vault is encrypted with a unique AES-256 key before it even touches disk. The key is never stored alongside the file in plaintext. Passwords are hashed with bcrypt. Tokens are signed with JWT. Files delete themselves.

The backend is written in Node.js, handles streaming encryption so even massive files don't blow memory, and auto-scales servers when traffic picks up. No third-party clouds. No analytics. No ads.

You get a unique URL. Anyone with the link (and passphrase, if set) can decrypt and view the file — until the timer runs out and it self-destructs.
TECH Encryption

Military-grade.
Zero compromises.

🔑
AES-256-CBC
The same standard used by governments and banks. Each file gets its own key — compromising one file reveals nothing about others.
32-byte key 16-byte IV crypto.randomBytes()
🧂
Bcrypt Hashing
File passwords aren't stored as-is. They're run through bcrypt with a salt factor of 10 — making brute-force attacks computationally impractical.
saltRounds: 10 bcrypt.hash() bcrypt.compare()
🪪
JWT Tokens
Authentication uses signed JSON Web Tokens — stateless, verifiable, and expiry-capable. No session state on the server side.
jsonwebtoken HS256 jwt.verify()
🌊
Stream Encryption
Files are piped directly through the cipher — never fully loaded into memory. This means even 2GB files encrypt without crashing the server.
createReadStream() .pipe(cipher) zero buffering

DATA Handling

What happens to
your file.

File received via Multer
The raw upload lands in uploads/ as a temp file with a safe, timestamp-prefixed name. Original filename is preserved in metadata only.
Streamed through AES cipher
A fresh key and IV are generated. The temp file is piped through createCipheriv into a .enc file. No full-file memory load.
Temp file wiped
Immediately after encryption finishes, fs.unlinkSync(tempPath) removes the plaintext original. Only the .enc remains.
Metadata stored as JSON
Key, IV, hashed password, original name, and expiry timestamp are saved in a sidecar .json file — never in a database.
Auto-delete on expiry
A background interval runs every 60 seconds, checking every .json expiry. When time's up, both .enc and .json are deleted. Gone forever.
